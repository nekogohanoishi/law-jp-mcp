## ツール設計

### `get_toc` — 目次取得
- **入力**: `lawId`（法令IDまたは法令名、略称可）
- **処理**: e-Gov APIのJSONツリー内にある`TOC`ノードを直接パース（APIが目次を構造化済み）
- **出力**: インデント付きの階層テキスト（タイトル＋条文範囲）

```

第一編　総則（第一条―第百八十八条の七）

  ...
第二編　第一審
  第三章　公判
    第二節　争点及び証拠の整理手続
      第一款　公判前整理手続（第三百十六条の二―第三百十六条の二十七）

        第三目　証拠開示に関する裁定（第三百十六条の二十五―第三百十六条の二十七）

ヒント: get_section で特定の編・章・節の条文を一括取得できます。
```

### `get_section` — 構造単位一括取得
- **入力**: `lawId`, `title`（タイトルテキスト、部分一致）, `maxChars`（デフォルト30000）
- **処理**: MainProvisionツリーを再帰探索し、タイトルが部分一致する構造ノードを特定 → 既存の`walkNode`で整形
- **出力**: 該当セクション内の全条文テキスト

タイトルマッチのルール:
- 構造接頭辞（「第一編　」等）を除去して比較
- `includes()` で部分一致
- 複数ヒット時は曖昧性解消メッセージを返す（パス一覧表示）
- 完全一致があればそれを優先

### このMCPの場合どれくらい使うか

1回の `get_section("民法", "不法行為")` の呼び出しで:
- 関数呼び出し: **1回**（100万回中の1回）
- CPU時間: 約 **0.5秒**（4時間 = 14,400秒中の0.5秒）
- 帯域: 約 **10KB**（100GB中の0.01MB）

### 各リソースの限界点

- サイトだけ: **月14万PV**で到達
- MCPだけ: **月28,800回**で到達

### 現実的な目安

| 使い方 | 超える？ |
|--------|----------|
| サイト月1,000PV + MCP 1日10回 | 余裕。**上限の2%以下** |
| サイト月5,000PV + MCP 1日50回 | 全く問題なし。**上限の10%程度** |
